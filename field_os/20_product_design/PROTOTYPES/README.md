# PROTOTYPES

## What lives here

**Prototypes** — interactive assemblies of wireframes used to feel and test a flow end to end. Fidelity is only as high as a test requires. Prototypes are throwaway design instruments, not production code and not implementation.

- **Purpose.** Make a flow tangible enough to feel its rhythm and put it in front of usability testing.
- **Inputs.** `WIREFRAMES/`, `USER_JOURNEYS/` (the flow being felt), `DESIGN_SYSTEM_INTEGRATION.md` (reused patterns).
- **Outputs.** Interactive prototypes per flow; each cites the journey and cards it assembles. These feed `USABILITY_TESTS/`.
- **Dependencies.** Accepted wireframes; the existing design system.
- **Responsible role.** Product Designer.
- **Relationship with the approved architecture.** Downstream. A prototype demonstrates an approved flow; it is not a build and creates no architectural fact. It must not simulate behavior the architecture forbids or imply data the model does not hold.
