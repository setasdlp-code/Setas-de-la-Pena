# Auditoría de diseño y funcionalidad

Fecha: 17 de julio de 2026  
Alcance: `simulador.html`, vista de escritorio 1280×720, vista móvil 390×844 y flujos de Inicio, Formulador, Optimizador, Bodega y Bitácora.

## Resultado ejecutivo

La app es visualmente coherente con Field OS, carga sin errores de ejecución propios y mantiene una jerarquía clara entre tareas. Los flujos centrales funcionan y el diseño responde sin desbordar el ancho de la página en móvil.

La auditoría detectó una contradicción operativa de severidad alta: el optimizador de Producción podía recomendar una receta con suplementación superior al límite y, después de cargarla, el sistema la mostraba como “Óptima” y “Apta” mientras el diagnóstico la marcaba como problema crítico. La corrección evita que los perfiles de bajo riesgo devuelvan esas recetas, alinea el veredicto con las advertencias y bloquea Producción ante fallas críticas. También se cerraron los bloqueadores técnicos de publicación: JSX precompilado, React production local y persistencia central con respaldo.

## Hallazgos y estado

| Prioridad | Hallazgo | Estado |
|---|---|---|
| Alta | El perfil Producción aceptaba 32,5–34,2% de suplemento con límite operativo de 20%. | Corregido: los perfiles `forceLowRisk` excluyen resultados sobre el límite. |
| Alta | “Apta / Producir normalmente” podía coexistir con problemas críticos. | Corregido: críticos y advertencias dominan el puntaje; Producción se bloquea ante estado crítico. |
| Alta | La penalización de EB usaba una condición lógicamente imposible y nunca se ejecutaba. | Corregido y protegido por prueba de regresión. |
| Media | Botones simbólicos `+`, `–`, `○` y `✕` no indicaban el ingrediente a tecnologías de asistencia. | Corregido en los controles esenciales del formulador. |
| Media | La marca superior era un `div` clicable y la navegación no declaraba página activa. | Corregido con botones semánticos, `aria-current` y landmark nombrado. |
| Media | Faltaban enlace de salto, foco visible reforzado y metadatos descriptivos. | Corregido. |
| Media | Etiquetas de navegación de 8,5 px y algunos controles de 9–11 px son pequeños para uso prolongado en campo. | Pendiente de una iteración visual dedicada; los objetivos táctiles principales sí alcanzan 44 px en móvil. |
| Media | La fila de filtros del formulador requiere desplazamiento horizontal en 390 px; el contenido de la página no se desborda. | Aceptado como patrón de carrusel, con degradado de pista existente. |
| Alta para publicación | React development y Babel transformaban JSX en el navegador mediante CDN. | Corregido: esbuild precompila JSX y empaqueta React production; todos los recursos de ejecución son locales. |
| Alta para operación | La persistencia residía únicamente en `localStorage`, sin respaldo. | Corregido para una instancia interna: API central, escritura atómica, respaldo de la versión anterior y migración única del estado histórico. |
| Alta para exposición pública | La API central no implementa autenticación, roles ni control de concurrencia multiusuario. | Pendiente; mantener el servidor en red confiable hasta incorporar identidad y base transaccional. |

## Pruebas ejecutadas

- Carga completa de la app y recursos locales.
- Navegación Inicio → Formulador → Bodega → Bitácora.
- Ejecución del optimizador automático y carga de la receta #1.
- Verificación de bodega sembrada: 4 ingredientes, 31 kg disponibles, 3 compras y 3 proveedores.
- Vista responsiva 390×844: sin desbordamiento horizontal del documento y barra inferior visible.
- Consola: sin errores propios y sin advertencia de Babel.
- Build local: JSX precompilado, React production, recursos sin CDN y CSP sin scripts inline.
- Persistencia: API de estado, recuperación tras reinicio y respaldo de la versión anterior.
- `npm run quality`: integridad local, invariantes de seguridad, accesibilidad esencial y pruebas automatizadas.

## Criterio de salida

La aplicación queda apta para pruebas internas controladas con el servidor Node incluido. No se recomienda exponerla a Internet ni usarla con múltiples operadores concurrentes hasta añadir autenticación, autorización y una base de datos transaccional. La operación desconectada no está implementada: si el servidor central cae, la sesión continúa en memoria con una alerta visible, pero sus cambios no se conservan.
