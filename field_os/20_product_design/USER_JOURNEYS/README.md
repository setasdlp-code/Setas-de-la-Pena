# USER_JOURNEYS

## What lives here

End-to-end **user journeys**: approved workflows staged as lived operator sequences across a real day — the operator's goal, context, steps, moments of attention, and emotional register from start to finish. Journeys set the narrative context that experience cards then decompose.

- **Purpose.** Turn approved workflows into lived sequences so design starts from real operator context, not screens.
- **Inputs.** `USER_WORKFLOWS.md` (the approved workflows), `PRODUCT_EXPERIENCE.md` (felt intent), `SYSTEM_INTERACTIONS.md` (what happens between actor and system).
- **Outputs.** Journey documents; each cites the workflow(s) it stages and the experience sections it serves. These feed `EXPERIENCE_CARDS/`.
- **Dependencies.** Approved workflows must exist first; `EXPERIENCE_BLUEPRINT.md` governs the flow.
- **Responsible role.** Product Designer.
- **Relationship with the approved architecture.** Downstream. Journeys stage approved workflows exactly as defined — they add narrative and attention, never new steps, entities, or workflows. A journey may not invent a path the architecture does not sanction.
